## When to Use This Skill

- A new field is added to `Character.cs` (e.g., `Tagline`, `CharacterType`,
  `ImageUrl`)
- API response shape for `/api/characters/{id}` changes
- Blazor component rendering rules change (e.g., conditional display, fallback
  text)
- Empty-state or null-fallback behavior is introduced or modified
- Partial data scenarios need coverage (nullable fields, missing relationships)
- Duplicate character records or conflicting rows may be influencing runtime
  behavior
- Seed data may no longer match what tests assume
- You want to catch likely regressions before merging

## Prerequisites

- Backend running or tests passing locally (`dotnet test`)
- Know which file(s) changed: model, controller, Blazor component, or DTO
- Access to `dotnet/Backend.Tests/` test project
- Workspace MCP configuration exists in `.vscode/mcp.json`
- The `fanhub-db` MCP server is available and trusted in Copilot Chat
- Access to the FanHub development database used by the MCP server

## Step 1 — Understand What Changed

Answer these questions before writing any tests:

1.  **What layer changed?**
    - [ ] `Character.cs` model (new field, changed type, new validation)
    - [ ] `CharactersController.cs` (new endpoint, changed response, status code)
    - [ ] `DTOs.cs` (new DTO property, response shape)
    - [ ] Blazor component (new conditional, fallback, rendering rule)
    - [ ] `SeedData` (new seed character, changed existing data)

2.  **Is the change additive or breaking?**
    - Additive: new optional field → lower regression risk
    - Breaking: field renamed/removed, type changed → high regression risk

3.  **Does the field have a default or is it nullable?**
    - `string?` nullable → needs null/empty fallback tests
    - `string` non-nullable with no `[Required]` → BUG territory (document it)
    - Has `[Required]` + `[StringLength]` → needs validation tests

4.  **Could live data be part of the problem?**
    - Duplicate characters with similar names
    - Optional fields populated inconsistently across rows
    - Seed data drift causing tests to pass with unrealistic assumptions
    - API and UI behavior depending on whichever row happens to load

Use the **Review Template** (`templates/review-template.md`) to capture your
answers.

## Step 2 — Check Live Data with MCP

Before deciding on tests, determine whether live FanHub data could be part of
the problem.

Use `#mcp-fanhub-db` when:

- duplicate character records are suspected
- nullable or optional fields behave inconsistently
- fallback UI may be hiding bad or partial data
- seed data or API responses may not match test fixtures
- a bug appears data-dependent instead of code-only

### What to look for

Inspect the real character records most likely to be affected and compare fields
such as:

- `Name`
- `Tagline`
- `CharacterType`
- `ImageUrl`
- `Bio`
- `Status`
- `ShowId`
- related `Quotes`
- related `Episodes`

### Questions to answer with MCP

1.  Are there duplicate or near-duplicate character rows?
2.  Which fields differ between those rows?
3.  Would those differences change API shape or Blazor rendering?
4.  Are current tests using fixture data that is cleaner than the real database?
5.  Does fallback behavior hide a real data-quality problem that should be
    surfaced?

### Capture these findings

- likely duplicate or conflicting rows
- fields that differ
- how those differences affect API responses, rendering, or regression risk
- what tests, cleanup work, or guardrails should happen before the PR

### Example MCP prompts

```text
@workspace #mcp-fanhub-db Find likely duplicate records for this character and
summarize which fields differ.
```

```text
@workspace #mcp-fanhub-db Compare the rows that could affect the character
detail page for this character and tell me which optional fields might trigger
fallback UI.
```

```text
@workspace #mcp-fanhub-db Show whether the real character data matches what our
tests currently assume for biography, image, quotes, and episodes.
```

If the exact schema differs from the examples above, adapt to the actual FanHub
tables and fields returned by the MCP server.

## Step 3 — Identify Tests to Add or Update

### Model Tests (`Backend.Tests/Models/CharacterTests.cs`)

| Change                               | Tests Needed                                             |
| ------------------------------------ | -------------------------------------------------------- |
| New required field added             | Validation passes with field; fails without field        |
| `[Required]` added to existing field | Validation now fails when field is                       |
| empty/null                           |
| `[StringLength(N)]` added            | Validation fails when string exceeds N chars             |
| New nullable field                   | No validation error when null; renders fallback          |
| Field removed                        | Existing tests referencing removed field must be updated |
| Duplicate-prevention rule added      | Validation fails for duplicate character                 |
| name within same show                |

### Controller Tests (`Backend.Tests/Controllers/CharactersControllerTests.cs`)

| Change                          | Tests Needed                                               |
| ------------------------------- | ---------------------------------------------------------- |
| New field in response           | `GetCharacter` response includes new field                 |
| New field required in POST body | `CreateCharacter` fails with 400 when field                |
| missing                         |
| New query filter                | `GetCharacters` filters correctly; returns empty list when |
| no match                        |
| Status code fix (200→201)       | Assert `CreatedAtActionResult` not                         |
| `OkObjectResult`                |
| Null check added                | `GetCharacter(999)` returns `404 NotFound` not exception   |
| Duplicate/conflicting data risk | Response shape remains stable even when                    |
| multiple similar rows exist     |

### Integration Tests (`Backend.Tests/Integration/`)

| Change                          | Tests Needed                                               |
| ------------------------------- | ---------------------------------------------------------- |
| DB schema change (new column)   | Migration applies; column reads/writes                     |
| correctly                       |
| New navigation property         | Related entity loads with `Include()`                      |
| Seed data updated               | Expected seeded characters present after `EnsureCreated()` |
|                                 |
| Duplicate cleanup or prevention | Seed/import path blocks or handles duplicate               |
| characters correctly            |

## Step 4 — Edge Cases Checklist

Before writing tests, check the **Edge Cases Checklist** in
`references/edge-cases-checklist.md`.

Key edge cases to always verify for character changes:

- [ ] Field is `null` → API returns field as `null`, not omitted
- [ ] Field is empty string `""` → treated differently from `null`?
- [ ] Field has max-length boundary value (e.g., exactly 200 chars for
      `Tagline`)
- [ ] Character with no related `Show` → null navigation property handled
- [ ] Character with no `Quotes` → empty array `[]` returned, not `null`
- [ ] Character with no `Episodes` → same as above
- [ ] `IsMainCharacter = false` → rendering handles both states
- [ ] `Status` is unexpected string value (e.g., `"unknown"`) → doesn't crash
- [ ] `ImageUrl` is `null` → fallback image or alt text shown
- [ ] Large `Bio` text → doesn't break layout or truncation
- [ ] Duplicate character rows differ in optional fields
- [ ] One duplicate row is more complete than another, but the wrong row may be
      surfaced
- [ ] Fallback UI hides a bad data-quality issue instead of making it obvious
- [ ] Test fixtures no longer reflect the real FanHub database state

## Step 5 — Generate or Revise Tests

### Priority Order

1.  **Regression tests first** — if field was renamed/removed, fix broken tests
    now
2.  **Happy path** — new field appears in response with valid data
3.  **Validation** — field rejects invalid input when attributes present
4.  **Null/empty fallback** — nullable fields don't crash consumers
5.  **Data-aware scenarios** — real duplicate or conflicting data patterns are
    represented in tests
6.  **Edge cases** — boundary values, unexpected states

### Test Naming Convention

Follow the existing pattern in the project:

```text
MethodName_Scenario_ExpectedResult
```

Examples:

```csharp
GetCharacter_WithTaglineField_ReturnsTaglineInResponse()
CreateCharacter_MissingTagline_Returns400WhenRequired()
GetCharacter_WithNullImageUrl_ReturnsNullNotOmitted()
Character_WithTaglineTooLong_FailsStringLengthValidation()
GetCharacter_WithDuplicateCharacterRows_HandlesConflictingOptionalDataSafely()
```

### Fixture Pattern

Use `DbContextFixture` with the `Seed()` helper for all controller tests:

```csharp
_fixture.Seed(ctx => {
    var show = new Show { Title = "Test Show", ... };
    ctx.Shows.Add(show);
    var character = new Character { ShowId = show.Id, Name = "Test", Tagline =
"Test tagline", ... };
    ctx.Characters.Add(character);
});
```

When MCP reveals risky live data patterns, update fixture data so tests reflect
the real FanHub scenarios rather than idealized rows.

See `references/test-patterns.md` for complete copy-paste examples.

## Step 6 — Recommend Cleanup or Guardrails

If MCP findings reveal that the bug is partly caused by the data, do not stop at
test recommendations.

Also recommend whether the PR needs any of the following:

- seed-data cleanup
- migration or constraint changes
- duplicate prevention rules
- fixture updates
- follow-up validation tests
- explicit PR notes about risky existing data

Examples:

- Merge or remove duplicate character rows in seed data
- Add validation to prevent duplicate character names within the same show
- Add regression tests for conflicting optional-field values
- Update fixtures so null, empty, and partial records mirror live data
- Add a follow-up task if the current PR should not include data cleanup
  directly
