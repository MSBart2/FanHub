# Documentation Refresh Workflow

**Goal**: Update FanHub documentation after meaningful code changes, keeping docs accurate and concise without rewriting unrelated sections.

---

## 📋 Overview

This prompt helps your team:

- Review recent code changes in the workspace/git history
- Identify which docs need updating (ARCHITECTURE.md, copilot-instructions.md)
- Make surgical, precise updates to only affected sections
- Verify documentation matches actual codebase
- Reuse this workflow for future changes

---

## 🔍 Phase 1: Identify Recent Changes

### Option A: Git-Based Discovery (If in Git repo)

1. **Run detection**:

   ```bash
   git diff HEAD~5..HEAD --name-only
   cd dotnet
   git log --oneline -10
   ```

2. **Look for**:
   - New Controllers or endpoints (`Backend/Controllers/*.cs`)
   - New Models or DTOs (`Backend/Models/*.cs`)
   - New Components (`Frontend/Components/*.razor`)
   - New/updated entity relationships (`Backend/Data/*.cs`)
   - Updated Program.cs (middleware, DI, configuration)
   - New intentional bugs documented

3. **Questions to ask**:
   - Are there new API endpoints?
   - Have Models been added or significantly changed?
   - Have new Components been added to the Frontend?
   - Has the technology stack changed (new NuGet packages)?
   - Are there new workflows or patterns?

### Option B: File System Discovery

Scan for recent modifications:

```bash
# Find recent files (modified in last N days)
Get-ChildItem -Path "c:\Users\rmathis\source\FanHub-1\dotnet\**" -Recurse |
  Where-Object { (Get-Date).AddDays(-7) -le $_.LastWriteTime } |
  Select-Object FullName, LastWriteTime | Sort-Object LastWriteTime
```

**Check these folders**:

- `dotnet/Backend/Controllers/` — API endpoints
- `dotnet/Backend/Models/` — Entity data models
- `dotnet/Frontend/Components/` — Razor components
- `dotnet/Backend/Program.cs` — Configuration, DI, middleware
- `dotnet/Backend/Data/` — DbContext, seed data
- `dotnet/BUGS.md` — Intentional bugs catalog

### Option C: User-Specified Changes

If prompted with argument, review the specified changes:

- "added character detail page" → Check CharacterDetail.razor, updated controller methods, DTO changes
- "updated validation" → Check Model attributes, validation error messages, feedback
- "new dependency" → Check .csproj files, Program.cs configuration

## 📐 Phase 2: Categorize Changes by Documentation Impact

Map changes to which docs need updating:

### ARCHITECTURE.md Updates Needed If:

- [ ] New controllers or major endpoint changes → Update **API Endpoints Reference** section
- [ ] New Models/Entities or relationship changes → Update **Data Models** section
- [ ] New Components or significant UI patterns → Update **Project Structure** section
- [ ] Build/run commands changed → Update **Build & Run Commands** section
- [ ] Technology stack changed (new NuGet packages) → Update **Technology Stack** table
- [ ] Code patterns/conventions established → Update **Code Patterns & Conventions** section
- [ ] New database migrations or schema changes → Update database sections
- [ ] Development workflow changed → Update **Development Workflow** section

### copilot-instructions.md Updates Needed If:

- [ ] New intentional bugs added → Add to **Intentional Bugs** section with explanations
- [ ] Code quality standards clarified/changed → Update **Code Quality Standards** section
- [ ] New features shipped (no longer "intentional bugs") → Move from bugs to **Active Features** section
- [ ] Setup or Quick Start process changed → Update **Quick Start** section
- [ ] Known bugs fixed → Remove from **Known Bugs** list
- [ ] New references or implementation patterns → Update **Implementation Reference** section
- [ ] Workshop learning objectives updated → Update **Learning Objectives** list

## 🔄 Phase 3: Surgical Documentation Updates

### Principle: Update Only What Changed

❌ **Don't**: Rewrite entire sections, change formatting, reorganize structure
✅ **Do**: Make minimal, targeted changes to factually incorrect or outdated information

### ARCHITECTURE.md Updates

#### A. API Endpoints Reference

**When**: New or modified controller endpoints

**Current Template** (verify what exists):

## ✅ Phase 4: Verification & Accuracy Check

After making doc updates, verify against actual code:

### For ARCHITECTURE.md

- [ ] **API Endpoints**: Run `dotnet run` and test with REST Client or Swagger UI (`http://localhost:5265/openapi/v1.json`)
- [ ] **Data Models**: Open `Backend/Models/Character.cs` and `DTOs.cs` — confirm field names/types match docs
- [ ] **Project Structure**: Run `tree dotnet/Frontend/Components/` — verify folder structure matches docs
- [ ] **Build Commands**: Actually run the commands in Terminal — do they work as documented?
- [ ] **Technology Stack**: Check `Backend.csproj` and `Frontend.csproj` — verify package versions match docs

### For copilot-instructions.md

- [ ] **Quick Start**: Follow steps exactly as written → does backend start on 5265? Does frontend start on 5171?
- [ ] **Code Quality Standards**: Pick a controller method — does it follow all listed standards?
- [ ] **Intentional Bugs**: If new bugs listed, can you find the exact code location in the codebase?
- [ ] **Active Features**: If status changed to "Completed", does the component actually render correctly?
- [ ] **Setup Guide**: Are all referenced files real? Do links point to existing docs?

## ⚠️ Common Mistakes to Avoid

❌ **Rewriting entire sections** — Only update what changed
❌ **Changing unrelated formatting** — Keep style/structure consistent
❌ **Stale links** — Verify all file references still exist
❌ **Inaccuracy** — Test commands before documenting them
❌ **Spreading changes across both docs** — Keep ARCHITECTURE.md structural, copilot-instructions.md for standards/bugs
❌ **Forgetting verification** — Run actual code to confirm docs are accurate
